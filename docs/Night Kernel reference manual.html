Section 0 - About the Night Kernel

The Night kernel has been built from the ground up as a full 32-bit drop-in replacement for the original 16-bit kernel of the FreeDOS operating system, and is intended to be loaded by the existing FreeDOS bootloader. Night was created with handcrafted assembly code using the NASM assembler, employs linear memory addressing, and operates in protected mode on the Intel x86 architecture, allowing users to enjoy protected mode abilities such as task switching between applications, protected memory, and increased overall performance in a DOS environment while maintaining full compatibility with their existing DOS applications.

Currently, the Night Kernel is a work in progress and is not yet suitable for everyday use; most of the code written to this point is providing internal functionality, and the kernel will not, at present, run user applications.





Section 1 - Building and Running the Kernel

Before we begin, keep in mind these directions are written based on a small variety of Linux Mint workstations used by the Night developers. This is not to say, however, that this is the only way to build the kernel; you can adapt the directions to whatever other compatible environment you like. Also, there are no pre-made libraries used in the creation of the Night Kernel; the repo contains all the code necessary to build the kernel from scratch without downloading any additional modules.

There are multiple ways to take Night for a test drive:



Booting from the pre-made VirtualBox image

In the builds folder of the repo, you will find a VirtualBox disk image named Night.vdi; a 16 MiB disk with a base-only install of FreeDOS 1.2 (sans sources) and the default kernel.sys swapped out for the Night Kernel. This disk image is fixed-size to allow for easier mounting under the host operating system and is ready to boot a VirtualBox virtual machine.

To create a virtual machine suitable for testing out Night, the default DOS configuration offered by VirtualBox can, for the most part, be used without alteration, however you may wish to name the machine Night for compatibility with the included build script, buildrun.sh. For details on the function of this script, see Building from source using buildrun.sh below.



Booting using the pre-assembled kernel binary

The repo's builds folder is also home to a pre-assembled kernel.sys, which you can copy over the existing FreeDOS kernel on any bootable FreeDOS medium, then just boot into FreeDOS from that bootable media as you normally would.



Building from source using buildrun.sh

If you're running in a Linux environment and have both VirtualBox and NASM installed, you can use the buildrun.sh script found in the root of the repo to streamline building and testing. It will automatically build the kernel sources into a binary image, mount the included VirtualBox disk image (mentioned above in Booting from the pre-made VirtualBox image) and then copy to that drive the kernel.sys binary which it just built. Afterwards, the script will launch the VirtualBox VM named Night, assuming you have already configured one. If you're not using VirtualBox, then the last step of the script will harmlessly fail, but the disk image will still be updated with the newly built kernel and be ready to use. Note that the script will prompt you for your root password as part of its normal function in mapping the vdi file to a loop device and mounting that device to a usable drive.

This method is used extensively by the Night developers to quickly test out new changes made to the kernel's source code on-the-fly during development, so if you're planning to contribute to the project, this just may be the option for you.



Building from source manually

Once you have the NASM assembler installed, the Night kernel can easily be built using the following command:

nasm -f bin -o builds/kernel.sys kernel.asm

Note that builds is the default directory to which the completed kernel image is set to be saved, but you may change this if you like.

After assembling, you will have a file called kernel.sys, which can be copied over top of the existing FreeDOS kernel on any bootable FreeDOS disk and subsequently run. If you don't want to assemble the kernel yourself, a pre-built kernel.sys is included in the builds folder for you.





Section 2 - The Night Kernel Driver Model

Drivers under Night are divided into two main categories:



Native drivers

A native driver is for a device attached to a bus, which has a built-in method of detection. Good examples of this would be PCI devices, USB devices, etc., which have an established, standardized interface through which they can be accessed. Night will probe the bus and load drivers if they match the devices it finds.



Legacy drivers

A legacy driver is for a device with no attachment to an established detection interface, such as older ISA or other onboard devices such as the PS/2 controller. For example, the kernel may not directly support the ISA bus, but devces attached to it may still be detected by testing the response received from certain I/O ports associated with the devices.



The most notable difference is that every legacy driver found by the kernel will have its init code run, whereas native drivers only get executed if a device is found which matches their capabilities.



Driver structure

Every Night driver, whether native or legacy, begins with a driver header, which is laid out as follows:

string signature
dword classMatch
dword subclassMatch
dword progIfMatch
dword driverFlags
dword WriteCodePointer
dword ReadCodePointer
To allow the driver to be found by the kernel, signature is always set to this series of bytes: 'N', 0x01, 'g', 0x09, 'h', 0x09, 't', 0x05, 'D', 0x02, 'r', 0x00, 'v', 0x01, 'r', 0x05.

The classMatch, subclassMatch, and progIfMatch fields are all used to match the driver to the device found on the PCI bus. A value of 0xFFFF in any of these fields means that the driver matches all values of that field; all other values are taken literally. For example, for the ATA driver, classMatch is set to 1, subClassMatch is set to 1, and ProgIfMatch is set to 0xFFFF, meaning that this driver can handle all devices with class 01 and subclass 01.

The driverFlags field is a bank of flags to identify attributes about this driver. Currently, the following bit definitions are supported:

Section 3 - Setting Options with ConfigBits

The Night Kernel offers configurable options which can be selected by altering certain bits in a specific value located within the kernel itself. Currently, the sole method of changing these options is to use a hex editor to locate the string ConfigBits in the kernel binary and changing the 32-bit value immediately following it. This allows options in the kernel to be changed without the need to recompile the kernel entirely. A utility will be created at later date which can automate selecting these options. At this point in time, the following options are supported:

 00  Enable debugging menu
 01  Verbose boot
 02  Use 50-line text mode instead of 25-line mode
